2 The `ListenerBus` issue occurred on the Spectrum jobs executed because of ListenerBus on a single thread which loops through all the Listeners for each event and processes each event synchronously.  This becomes a bottleneck for large jobs and can cause events to get dropped. SparkListener is a mechanism in Spark to intercept events from the Spark scheduler that are emitted over the course of execution of a Spark application.  Spark relies on SparkListeners internally to manage communication between internal components in the distributed environment for a Spark application, e.g. web UI, event persistence (for History Server), dynamic allocation of executors, keeping track of executors (using HeartbeatReceiver) and others.

 

The problem is that the event processor in `ListenerBus` is a single thread which loops through all the Listeners for each event and processes each event synchronously.  This single threaded processor becomes a bottleneck for large jobs and can cause events to get dropped which can lead to metrics reported erronously and/or job failures. 

 

To mitigate this issue, we need to ensure the framework can handle the number of concurrent events for the application so that event don't get drop and impact application execution		 Error: scheduler.LiveListenerBus: Dropping SparkListenerEvent because no remaining room in event queue  spark.scheduler.listenerbus.eventqueue.size > 10K example spark.scheduler.listenerbus.eventqueue.size = 300000
